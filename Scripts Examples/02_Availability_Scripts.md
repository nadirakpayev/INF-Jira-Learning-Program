# Availability Scripts

Источник:
- `Source Docs/Forte-J-ITSM25+Руководство+администратора_v1.0.docx`

## `Назначение AM manager`

- Тип: field-setting script
- Назначение: выставить фиксированное значение `AM manager`
- Особенность: захардкоженный ключ `AD-33691`

## `Назначение Исполнителя заявки`

- Тип: assignment script
- Назначение: назначить исполнителя по ИТ-владельцу системы из Assets
- Основная логика:
  - прочитать поле `Система`;
  - найти в Assets ИТ-владельца;
  - найти у сотрудника `Jira Пользователь`;
  - назначить его исполнителем

## `Рассылка для ИТ-владельца`

- Тип: notification script
- Назначение: отправить email ИТ-владельцу о завершении события недоступности
- Особенности:
  - фильтрация неактивных пользователей по статусу `15`;
  - использование `Фактическая длительность простоя`

## `Расчет плановой длительности простоя (в минутах)`

- Тип: calculated field script
- Назначение: рассчитать плановую длительность простоя
- Условие:
  - `Тип недоступности (AM)` должен быть `Плановая`

## `Расчет фактической длительности простоя (в минутах) для внеплановой`

- Тип: calculated field script
- Назначение: рассчитать фактическую длительность только для внеплановой недоступности

## `Расчет фактической длительности простоя (в минутах)`

- Тип: calculated field script
- Назначение: универсальный расчет фактической длительности без проверки типа недоступности

## `Создание события недоступности из ЗИ`

- Тип: issue creation script
- Назначение: создать задачу `Событие недоступности` из ЗИ
- Делает:
  - переносит систему, модуль, фактические даты;
  - получает `Вес модуля` из Assets;
  - нормализует тип простоя и процент недоступности;
  - создает связанную задачу в проекте `FORTE`

## `Условия поля AM manager`

- Тип: workflow condition
- Назначение: разрешить действие, если текущий пользователь — `AM manager` задачи или администратор
- Условия:
  - поле `AM manager` должно содержать `AD-...`;
  - сотрудник ищется в Assets по атрибуту `Jira Пользователь`

## Общие наблюдения

- Почти все скрипты по доступности завязаны на Assets-объекты `Система`, `Сотрудник`, `Модули услуг`.
- Используются жесткие ключи и строковый парсинг значений полей.
- Логика ориентирована на Jira Data Center и серверные post-function / condition сценарии.
