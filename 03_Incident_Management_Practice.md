# Практическое задание 3. Процесс управления инцидентами

## Ситуация

Нужно развернуть процесс инцидентов для быстрого восстановления сервиса. Процесс должен поддерживать маршрутизацию по линиям поддержки, запрос дополнительной информации, передачу вендору, фиксацию решения и закрытие с подтверждением результата.

## Цель

Настроить Cloud-процесс инцидентов, пригодный для ежедневной операционной работы.

## Роль в программе

Это второе сквозное задание блока 1. Оно использует каталог услуг и `Components`, созданные в предыдущем модуле.

## Что нужно сделать

1. Создайте request type или issue type для инцидентов.
2. Настройте lifecycle со статусами:
   - Открыт;
   - Назначен на 1-ю линию;
   - Назначен на 2-ю линию;
   - Назначен на 3-ю линию;
   - В работе на 1-й линии;
   - В работе на 2-й линии;
   - В работе на 3-й линии;
   - В работе у вендора;
   - Запрос доп. информации;
   - Смена линии;
   - Решен;
   - Закрыт.
3. Определите обязательные поля:
   - категория;
   - услуга;
   - component;
   - модуль;
   - связанные КЕ;
   - приоритет;
   - канал поступления;
   - группа исполнителей;
   - координатор;
   - исполнитель;
   - решение;
   - описание решения.
4. Настройте формы:
   - регистрация через портал;
   - регистрация агентом;
   - решение инцидента;
   - закрытие;
   - передача вендору;
   - запрос доп. информации.
5. Реализуйте автоматику маршрутизации:
   - по услуге;
   - по component;
   - по КЕ;
   - по линии поддержки.
6. Настройте переходы для смены линии и ошибочного назначения.
7. Реализуйте базовые уведомления:
   - о назначении;
   - о запросе информации;
   - о решении;
   - о просрочке SLA.
8. Настройте SLA:
   - первая реакция;
   - время решения;
   - отдельные цели по приоритетам.
9. Подготовьте очереди:
   - новые инциденты;
   - по линиям поддержки;
   - у вендора;
   - ожидает ответа заявителя;
   - инциденты с высоким приоритетом;
   - нарушенные SLA.
10. Выполните 4 теста:
   - типовой инцидент 1-й линии;
   - эскалация на 2-ю линию;
   - эскалация на 3-ю линию и вендору;
   - закрытие после подтвержденного решения.

## Cloud-акцент

- В Cloud лучше делать управляемую маршрутизацию через automation и очередь, а не через десятки разветвленных переходов.
- Components используйте для систем, сервисов или доменов ответственности, а модули и КЕ моделируйте через select list.
- Для метрик используйте native SLA Jira Service Management.

## Артефакты

- Процесс инцидентов в service project.
- Карта маршрутизации.
- Очереди и SLA.
- Тестовые сценарии с результатами.

## Критерии приемки

- Инцидент проходит полный цикл от регистрации до закрытия.
- По карточке задачи видно, кто отвечает за инцидент сейчас и на какой линии он находится.
- SLA и очереди позволяют оперативно управлять инцидентным потоком.
