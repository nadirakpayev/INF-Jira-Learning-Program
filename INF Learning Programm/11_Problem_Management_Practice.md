# Практическое задание 11. Процесс управления проблемами

## Ситуация

Повторяющиеся инциденты и значительные сбои должны приводить к формализации проблем, диагностике корневой причины, фиксации known error и, при необходимости, запуску изменения.

## Цель

Настроить Cloud-процесс problem management с привязкой к инцидентам, known error и RFC.

## Роль в программе

Это второе процессное задание блока 3. Оно должно использовать результаты major incident и готовить основу для change management.

## Что нужно сделать

1. Создайте issue type для проблемы.
2. Настройте lifecycle со статусами:
   - Кандидат в проблему;
   - Первичная диагностика;
   - Полная диагностика;
   - В работе у вендора;
   - Известная ошибка;
   - Применение решения;
   - Выполнено;
   - Закрыто;
   - Отмена.
3. Определите роли:
   - инициатор;
   - менеджер проблемы;
   - команда диагностики;
   - координатор проблемы.
4. Настройте поля:
   - component;
   - модуль;
   - связанные инциденты;
   - команда диагностики;
   - координатор;
   - известная ошибка;
   - обходное решение;
   - постоянное решение;
   - ссылка на RFC;
   - ссылка на статью базы знаний.
5. Реализуйте сценарии создания:
   - вручную агентом;
   - из значительного инцидента;
   - из инцидента-кандидата.
6. Настройте автоматику:
   - уведомление менеджеров о новом кандидате;
   - заполнение команды диагностики по системе;
   - перевод связанных инцидентов в режим ожидания решения проблемы;
   - возврат инцидентов из парковки после выполнения или отмены;
   - автоматическое закрытие проблемы через установленный срок после выполнения.
7. Спроектируйте Cloud-вариант known error:
   - через отдельный тип записи, статью базы знаний или controlled custom field с ссылкой на проблему;
   - при необходимости через select list с ограниченным набором известных ошибок для учебного контура.
8. Реализуйте связь с RFC:
   - из статуса применения решения должна создаваться или связываться заявка на изменение.
9. Настройте SLA:
   - время на диагностику;
   - время на устранение;
   - время работы с вендором.
10. Проведите 3 теста:
   - проблема с обходным решением;
   - проблема с переходом в RFC;
   - проблема, возвращенная инициатором на доработку.

## Cloud-акцент

- Если в исходной серверной логике была глубокая работа со связанными инцидентами, в Cloud реализуйте ее через понятные automation-правила и четкий набор связанных статусов.
- Не усложняйте known error: для учебного задания допустим управляемый вариант на issue links, select list и статье базы знаний.

## Артефакты

- Настроенный процесс problem management.
- Схема взаимодействия с инцидентами и RFC.
- Cloud-описание того, как ведется known error database.

## Критерии приемки

- Проблема может быть создана из инцидентного контекста.
- У процесса есть понятный диагностический и исполнительский цикл.
- Система показывает, как обходное решение и постоянное решение влияют на связанные инциденты.
