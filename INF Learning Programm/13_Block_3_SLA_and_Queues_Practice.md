# Практическое задание 13. SLA и очереди для расширенных ITSM-процессов

## Ситуация

Во втором блоке процессы становятся сложнее: появляются значительные инциденты, проблемы и изменения. Без продуманного набора очередей и SLA команда теряет управляемость, а руководитель не видит узкие места.

## Цель

Настроить SLA и очереди для процессов блока 3 так, чтобы можно было управлять эскалациями, диагностикой, изменениями и контрольными точками выполнения.

## Роль в программе

Это завершающее прикладное задание блока 3 перед финальным workshop. Оно собирает major incident, problem и change в единый контрольный контур.

## Что нужно сделать

1. Определите SLA для значительных инцидентов:
   - время назначения менеджера;
   - время до восстановления;
   - общее время до решения.
2. Определите SLA для проблем:
   - время до первичной диагностики;
   - время диагностики;
   - время до применения решения;
   - время до закрытия после выполнения.
3. Определите SLA для RFC:
   - подготовка изменения;
   - выполнение;
   - приемка инициатором.
4. Для каждого SLA зафиксируйте:
   - условия старта;
   - статусы паузы;
   - условия остановки;
   - варианты целей по типу или приоритету.
5. Настройте очереди по значительным инцидентам:
   - активные;
   - на подтверждении;
   - в восстановлении;
   - просроченные.
6. Настройте очереди по проблемам:
   - кандидаты;
   - первичная диагностика;
   - полная диагностика;
   - у вендора;
   - известная ошибка;
   - просроченные.
7. Настройте очереди по изменениям:
   - на согласовании;
   - запланированные;
   - в выполнении;
   - в тестировании;
   - неуспешные;
   - ожидающие приемки.
8. Настройте представление очередей так, чтобы были видны:
   - тип задачи;
   - статус;
   - component;
   - исполнитель;
   - ключевые даты;
   - индикаторы SLA.
9. Проведите тестовую проверку на трех сценариях:
   - значительный инцидент;
   - проблема с переходом в known error или применение решения;
   - RFC с полным циклом от согласования до приемки.

## Cloud-акцент

- Не пытайтесь перегрузить проект десятками отдельных SLA-метрик. Оставляйте только те, которые реально помогают управлять процессом.
- Очереди должны быть полезны и агенту, и руководителю процесса.
- Там, где полное покрытие SLA избыточно, зафиксируйте это как сознательное решение архитектуры курса.

## Артефакты

- Набор SLA по major incident, problem и change.
- Набор рабочих очередей по процессам блока 3.
- Результаты тестовой проверки.

## Критерии приемки

- По очередям видно, где находится поток работ и где есть риски.
- SLA дают контроль по ключевым точкам расширенных процессов.
- Процессы блока 3 стали управляемыми не только логически, но и операционно.
