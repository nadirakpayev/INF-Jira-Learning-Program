# Практическое задание 2. Каталог услуг и модель запросов

## Ситуация

Перед настройкой процессов нужно сформировать учебный каталог услуг. Он будет использоваться в запросах на обслуживание, инцидентах, изменениях и аналитике. Каталог должен быть понятен заявителю и пригоден для маршрутизации обращений.

## Цель

Собрать минимально жизнеспособный каталог услуг, пригодный для дальнейшей настройки workflow и маршрутизации.

## Роль в программе

Это ключевое сквозное задание блока 1. Его результат используется в инцидентах, сервисных запросах и в финальном workshop блока.

## Что нужно сделать

1. Сформируйте структуру категорий услуг.
2. Для каждой категории определите набор услуг, которые будут использоваться в учебной среде.
3. Для каждой услуги зафиксируйте:
   - наименование;
   - описание;
   - тип обращения;
   - связанный сервис или система;
   - статус активности;
   - предполагаемую группу поддержки.
4. Создайте в проекте `Components`, где каждый component отражает систему, сервисный домен или зону ответственности.
5. Для каждого component настройте:
   - понятное название;
   - owner или lead;
   - default assignee, если такой вариант используется в вашей модели;
   - описание зоны ответственности.
6. Подготовьте не менее 12 услуг, покрывающих разные сценарии:
   - сообщение о проблеме;
   - доступы;
   - учетные записи;
   - консультации;
   - оборудование;
   - телеком;
   - установки и настройки.
7. Спроектируйте, как пользователь будет выбирать услугу:
   - через request type;
   - через каскадную логику категория -> услуга;
   - через портал с понятными названиями.
8. Определите, какие элементы каталога будут отражаться в Jira через `Components`, а какие через `select list`.
   - рекомендуемый подход: Components для систем, сервисов или зон ответственности;
   - select list для модулей, видов услуг и вспомогательных справочников.
9. Определите правила формирования темы заявки:
   - для инцидентов;
   - для сервисных запросов;
   - для RFC.
10. Подготовьте реестр связей:
   - категория услуги -> услуга;
   - услуга -> тип обращения;
   - услуга -> component;
   - услуга -> группа поддержки.
11. Подготовьте Cloud-вариант каталога только на базе стандартных возможностей:
   - request type;
   - Components;
   - select list;
   - forms;
   - таблица соответствий вне Jira.

## Cloud-акцент

- В Cloud удобнее проектировать каталог так, чтобы им можно было пользоваться и на портале, и в агентском интерфейсе.
- Для учебного стенда не перегружайте портал десятками request type. Лучше показать продуманную структуру и правила масштабирования.
- Components должны помогать маршрутизации и аналитике, а не дублировать все справочники сразу.
- Если используете default assignee на уровне component, зафиксируйте правило, когда заявка идет на component lead, а когда остается на проектном назначении.

## Артефакты

- Каталог категорий и услуг.
- Карта соответствий услуга -> тип обращения -> группа поддержки.
- Черновик структуры portal/request type.

## Критерии приемки

- Пользователь может выбрать услугу без знания внутренней структуры поддержки.
- Каталог пригоден для автоматической маршрутизации в следующих модулях.
- Структура не привязана к исходным названиям организаций и выглядит как самостоятельный учебный продукт.
