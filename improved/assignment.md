# Программа задания для кандидата

## Цель

Показать, что вы способны:
- освоить базовую работу в `Jira Service Management Cloud`;
- последовательно строить процессы от простого к сложному;
- различать, что должно жить в `Jira`, а что должно жить во внешней модели данных;
- доводить решение до проверяемого результата, а не только до устного описания.

## Общая логика программы

Программа состоит из `4` блоков.

Важно:
- `Блок 1` и `Блок 2` используются для решения, подходит ли кандидат;
- `Блок 3` и `Блок 4` выполняются только после положительного решения и нужны для подготовки к работе в компании;
- `Source Docs` не являются частью задания и не требуются для выполнения.

## Общие ограничения

Обязательно:
- использовать `Jira Service Management Cloud`;
- использовать `company-managed project`;
- использовать `Components` как представление систем в Jira;
- использовать `select list` там, где не нужен отдельный объект;
- использовать локальный `PostgreSQL` и `pgAdmin` без контейнеров;
- использовать выданные тестовые данные `CMDB`.

Запрещено:
- использовать `Assets` и trial-функции;
- использовать `Docker` или иные контейнеры;
- заменять выданный датасет произвольными данными;
- сдавать только устное объяснение без артефактов;
- трактовать `Source Docs` как дополнительный скрытый backlog требований.

## Блок 1. Jira SM Cloud Foundation

### Что нужно сделать

1. Создать собственную среду `Atlassian Cloud`.
2. Развернуть один `Jira Service Management` проект.
3. Создать базовый сервисный каталог.
4. Настроить процесс `Incident`.
5. Настроить процесс `Service Request`.
6. Настроить `SLA` и `Queues` для процессов блока.

### Обязательный минимум

- собственный cloud-site;
- один `company-managed service project`;
- минимум `3` `Components`, где `Component = система`;
- маршрутизация через `Components` и default assignee там, где это возможно без `Assets`;
- минимум `6` услуг в каталоге;
- минимум `2` request types:
  - `Incident`;
  - `Service Request`;
- рабочий процесс `Incident`;
- рабочий процесс `Service Request`;
- минимум по `SLA`:
  - first response;
  - resolution / fulfillment;
- минимум по `Queues`:
  - для `Incident` не менее `3`;
  - для `Service Request` не менее `3`.

### Что проверяется в блоке

- умеете ли вы самостоятельно поднять среду;
- понимаете ли вы базовые сущности `JSM Cloud`;
- можете ли вы собрать минимально полезный operational core;
- умеете ли вы строить каталог и маршрутизацию без лишней сложности.

## Блок 2. Data and Process Core

### Что нужно сделать

1. Взять выданные тестовые данные `CMDB`.
2. Спроектировать реляционную модель данных.
3. Развернуть локальную БД в `PostgreSQL` через `pgAdmin`.
4. Реализовать модель в БД.
5. Загрузить тестовые данные.
6. Настроить упрощенную связку:
  - `Major Incident`;
  - `Problem`;
  - `Change`.
7. Настроить `SLA` и `Queues` для второго блока.

### Обязательный минимум

- использовать выданный `CMDB`-датасет;
- подготовить модель сущностей и связей;
- создать локальную БД без контейнеров;
- создать таблицы, ключи и связи;
- загрузить тестовые данные;
- подготовить проверочные SQL-запросы;
- показать, как данные из модели соотносятся с данными в `Jira`;
- показать сквозную цепочку `Major Incident -> Problem -> Change`;
- настроить отдельные `SLA` и `Queues`, relevant для задач второго блока.

### Что проверяется в блоке

- понимаете ли вы разницу между процессом и моделью данных;
- умеете ли вы реализовать простую реляционную схему руками;
- умеете ли вы строить связанную процессную логику, а не только отдельные формы;
- умеете ли вы работать в ограничениях `Jira SM Cloud`, не имитируя `Assets`.

## Решение по кандидату

После проверки `Блока 1` и `Блока 2` принимается решение:
- `PASS` — кандидат подходит и переходит к следующим блокам;
- `FAIL` — кандидат не продолжает программу.

`Блок 3` и `Блок 4` не используются, чтобы компенсировать слабый результат в первых двух блоках.

## Блок 3. Availability Management

Этот блок выполняется только после `PASS`.

### Что нужно сделать

- смоделировать и настроить базовый контур `Availability Management` в `Jira SM Cloud`;
- показать связь недоступности с сервисами, системами и окнами обслуживания;
- настроить `SLA` и `Queues`, которые помогают сопровождать этот процесс.

### Цель блока

- закрепить работу с процессами в `Cloud`;
- проверить, как кандидат переносит уже изученные паттерны на новый процесс.

## Блок 4. Integration Model Design

Этот блок выполняется только после `PASS`.

### Что нужно сделать

- спроектировать модели будущих интеграций;
- описать источники, приемники, сущности, точки обмена и минимальные контракты данных;
- не реализовывать фактические интеграции в коде;
- настроить итоговый демонстрационный пакет по блоку.

### Цель блока

- подготовить кандидата к реальным проектным задачам;
- закрепить умение проектировать, а не только настраивать формы и workflow.

## Ожидаемый результат

На screening-этапе ожидается:
- проверяемый `JSM Cloud` проект;
- работающий каталог;
- `Incident` и `Service Request`;
- `SLA` и `Queues`;
- рабочая локальная `PostgreSQL`-модель `CMDB`;
- понятная связка `Major Incident -> Problem -> Change`.

На onboarding-этапе ожидается:
- перенос тех же принципов на `Availability`;
- проектирование интеграционной модели;
- более уверенная работа с платформенными ограничениями.

## Формат сдачи

Сдача идет по блокам и должна соответствовать `submission_format.md`.

Без обязательных артефактов блок считается несданным.
