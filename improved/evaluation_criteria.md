# Правила оценки

## 1. Общая модель

Используются два разных режима оценки:
- screening-оценка для `Блока 1` и `Блока 2`;
- completion-оценка для `Блока 3` и `Блока 4`.

Только screening-оценка влияет на решение по кандидату.

## 2. Screening: PASS / FAIL

Кандидат получает `PASS`, если одновременно выполняются все условия:
- сдан `Блок 1`;
- сдан `Блок 2`;
- предоставлен пакет артефактов по `submission_format.md`;
- по screening-рубике набрано не менее `65/100`;
- отсутствуют критические ошибки.

Кандидат получает `FAIL`, если выполнено хотя бы одно условие:
- не сдан `Блок 1` или `Блок 2`;
- отсутствуют обязательные артефакты;
- использованы запрещенные инструменты;
- решение нельзя проверить без длительного дополнительного интервью;
- не показано понимание границы между `Jira` и внешней моделью данных.

## 3. Правила сдачи Блока 1

`Блок 1` считается сданным, если:
- подтвержден собственный `Atlassian Cloud` site;
- подтвержден один `company-managed JSM project`;
- созданы и осмысленно используются `Components`;
- есть каталог с минимальным обязательным объемом;
- работают `Incident` и `Service Request`;
- есть `SLA` и `Queues` для обоих процессов;
- есть evidence, позволяющий проверить конфигурацию без live-рассказа.

`Блок 1` не сдан, если:
- нет работающей среды;
- `Components` существуют только формально;
- процессы не связаны с каталогом;
- отсутствуют `SLA` или `Queues`;
- виден только happy-path без маршрутизации и контроля.

## 4. Правила сдачи Блока 2

`Блок 2` считается сданным, если:
- использован выданный `CMDB`-датасет;
- есть модель сущностей и связей;
- есть локальная `PostgreSQL` без контейнеров;
- создана рабочая схема с `PK` и `FK`;
- загружены тестовые данные;
- есть SQL-проверка модели;
- показана связка `Major Incident -> Problem -> Change`;
- есть `SLA` и `Queues`, относящиеся к задачам блока;
- объяснено, что хранится в `Jira`, а что остается во внешней модели.

`Блок 2` не сдан, если:
- нет рабочей локальной БД;
- нет ключей и связей;
- датасет подменен;
- вместо модели есть только narrative-описание;
- процессная связка существует только на словах;
- нет доказуемой связи между моделью данных и настройкой Jira.

## 5. Completion-оценка для Блока 3 и Блока 4

После `PASS` используются статусы:
- `Completed`;
- `Completed with comments`;
- `Needs revision`.

Эти блоки:
- не меняют screening-решение задним числом;
- не используются для компенсации провала в `Блоке 1` или `Блоке 2`;
- нужны для подготовки к реальной работе.

## 6. Критические ошибки

Автоматический `FAIL`:
- использование `Assets`;
- использование `Docker` или контейнеров для БД;
- отсутствие обязательных артефактов;
- подмена результата скриншотами без логики или логикой без доказательств;
- сдача только устным рассказом;
- отсутствие рабочей локальной модели данных;
- явное несоответствие между описанием и evidence.

## 7. Допустимые ошибки

Не ведут к автоматическому `FAIL`, но снижают балл:
- избыточные статусы;
- спорный naming;
- небольшая денормализация;
- одно manual-решение вместо автоматизации при честно описанном ограничении;
- частично слабая визуальная чистота submission-пакета.

## 8. Сильный кандидат

Признаки:
- аккуратно ограничивает scope;
- строит решение от потока работы, а не от количества настроек;
- делает простую, но рабочую модель данных;
- объясняет ограничения платформы спокойно и точно;
- дает артефакты, которые можно проверить без длинного интервью.

## 9. Слабый кандидат

Признаки:
- компенсирует отсутствие результата длинным текстом;
- смешивает данные `Jira` и внешнюю модель;
- не умеет упростить решение;
- строит несвязанные процессы;
- не может доказать, что конфигурация реально работает.

## 10. Что можно автоматизировать

Можно автоматизировать:
- проверку структуры `submission/`;
- наличие обязательных файлов по блокам;
- наличие SQL-файлов;
- наличие `PK`, `FK`, `CREATE TABLE`, `INSERT` и проверочных `SELECT`;
- наличие ссылок на обязательные evidence-артефакты в `README.md`;
- наличие обязательных summary-файлов по `SLA`, `Queues`, `Automation`;
- использование кодов и имен из выданного `CMDB`-датасета.

## 11. Что остается на ревьюера

Ревьюер вручную оценивает:
- насколько осмысленно выбраны `Components` и маршруты;
- насколько логична схема данных;
- насколько убедительно разведены `Jira` и внешняя модель;
- есть ли признаки реального понимания, а не механической сборки.
