# Античит-заметки

## Основные риски

- кандидат присылает красивое описание вместо работающего результата;
- кандидат проходит `Блок 1` за счет шаблонной конфигурации без понимания;
- кандидат присылает SQL, который не выполнялся;
- кандидат пытается компенсировать слабый `Блок 2` сильным устным рассказом;
- кандидат использует `Source Docs` как аргумент для ухода от формулировок задания.

## Меры снижения риска

1. Проверять только staged submission по `submission_format.md`.
2. Оценивать `Блок 1` и `Блок 2` независимо друг от друга.
3. Не принимать длинную демонстрацию как замену артефактам.
4. Сверять:
   - названия `Components`;
   - названия услуг;
   - поля и статусы;
   - сущности и коды из `CMDB`-датасета;
   - SQL и evidence из `pgAdmin`.
5. Спрашивать короткие верификационные вопросы только по конкретным артефактам.

## Быстрые признаки поверхностного прохождения

- в `Jira` и в SQL разные сущности и термины;
- `Components` есть, но они не участвуют в маршрутизации;
- `SLA` и `Queues` описаны, но не показаны;
- есть ER-модель, но нет рабочей схемы БД;
- есть story про процессы, но нет evidence связки `Major Incident -> Problem -> Change`;
- кандидат ссылается на `Source Docs` вместо выполнения прямых требований задания.
