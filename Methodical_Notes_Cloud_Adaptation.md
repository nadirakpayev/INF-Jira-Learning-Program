# Методические заметки. Правила адаптации требований под Jira Cloud

## Ситуация

Исходные материалы описывают процессы с логикой, характерной для серверного/дата-центрового подхода: сложные скрипты, UI-behaviour, кастомные post-function, отдельные плагины для SLA и глубокая зависимость от серверной автоматизации. В Cloud часть этой логики нужно реализовывать иначе.

## Цель

Подготовить единый набор архитектурных правил, по которым вы будете адаптировать все последующие задания.

## Что нужно сделать

1. Подготовьте список возможностей, которые в Cloud должны стать базовым стандартом реализации:
   - request types;
   - forms;
   - native approvals;
   - queues;
   - dashboards and filters;
   - native SLA;
   - Automation for Jira;
   - Components;
   - select list и другие стандартные custom fields.
2. Для каждой серверной практики определите Cloud-замену:
   - behaviours и сложная логика форм -> forms, field configuration, request type logic, automation;
   - listeners и часть post-function -> automation rules;
   - job-логика -> scheduled automation;
   - Time-to-SLA и похожая метрика на уровне JSM -> native SLA, date fields, dashboards;
   - серверные справочники -> Components, select list, mapping table вне Jira или комбинация этих подходов.
3. Зафиксируйте, какие требования вы реализуете нативно, а какие переводите в упрощенный вариант.
4. Составьте правила проектирования:
   - один основной service project для интегрированных процессов или несколько проектов по четкой причине;
   - только нейтральные названия;
   - одинаковый подход к статусам и именованию automation;
   - отдельное соглашение по ролям, группам и очередям.
5. Составьте список Cloud-ограничений, которые нужно учитывать во всех модулях:
   - не вся серверная логика форм переносится один в один;
   - часть сложных проверок выполняется после события через automation, а не до сохранения;
   - в программе не используется Assets и не предполагается активация trial;
   - Components работают на уровне проекта, поэтому их модель нужно проектировать экономно;
   - справочники со сложными связями между сущностями не нужно насильно встраивать в Jira;
   - массовые автоматизации и пересчеты нужно проектировать экономно;
   - часть BI-сценариев потребует экспорт, стороннюю аналитику или отдельную витрину.
6. Зафиксируйте модель принятия решений:
   - сначала стандартные возможности Jira Cloud;
   - затем Components или select list;
   - затем automation;
   - затем допустимое упрощение;
   - затем явное описание gap и плана закрытия.

## Результат задания

Подготовьте короткий документ или страницу с тремя разделами:

- `Что переносим без изменений`
- `Что реализуем иначе в Cloud`
- `Что требует отдельного решения или ограничения scope`

## Критерии приемки

- Для каждого следующего модуля можно быстро понять, какой механизм Cloud использовать.
- Нет прямой зависимости от серверных скриптов.
- Все спорные места помечены заранее, а не обнаруживаются в конце настройки.

## Подсказка

Если требование невозможно красиво реализовать нативно, не пытайтесь скрыть этот факт. Для учебной программы ценнее честно показать границу платформы и предложить рабочий Cloud-вариант.
