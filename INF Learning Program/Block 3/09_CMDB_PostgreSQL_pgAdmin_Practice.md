# Практическое задание 9. Локальная PostgreSQL и реализация CMDB-модели через pgAdmin

## Ситуация

После проектирования CMDB как модели данных участнику нужно сделать следующий шаг: не просто нарисовать сущности и связи, а перенести их в реальную локальную базу данных. Задание должно оставаться простым и понятным для джунов, поэтому используется локальная PostgreSQL без контейнеров и настройка через pgAdmin.

## Цель

Развернуть локальную PostgreSQL, подключить ее к pgAdmin и реализовать в ней CMDB-модель, спроектированную в предыдущем задании.

## Роль в программе

Это второе задание блока 3. Оно идет сразу после моделирования CMDB и переводит абстрактную схему в реальную учебную БД.

Если не хватает базы по `PostgreSQL`, `pgAdmin`, таблицам, ключам, `SELECT`, `JOIN` или подзапросам, сначала или параллельно откройте [DataBase Fundamentals/README.md](../DataBase%20Fundamentals/README.md). Для этого задания такой трек становится `обязательным`, если без него вы не можете создать и проверить учебную БД самостоятельно.

## Что нужно сделать

1. Установите PostgreSQL локально на свой компьютер без использования Docker или других контейнеров.
2. Установите pgAdmin и подключитесь к локальному экземпляру PostgreSQL.
3. Создайте отдельную учебную базу данных для CMDB.
4. Создайте схему таблиц на основе модели из задания [08_CMDB_Data_Modeling_Practice.md](08_CMDB_Data_Modeling_Practice.md).
5. Используйте данные из файла [CMDB_Test_Data_Reference.md](../Block%202/CMDB_Test_Data_Reference.md) как источник тестовых записей.
6. Реализуйте в PostgreSQL минимум следующие таблицы:
   - services;
   - systems;
   - modules;
   - configuration_items;
   - software;
   - servers;
   - licenses;
   - workgroups;
   - employees;
   - maintenance_windows.
7. Для каждой таблицы определите:
   - первичный ключ;
   - обязательные поля;
   - внешние ключи;
   - ограничения целостности там, где они нужны.
8. Если по вашей модели есть связи многие-ко-многим, создайте отдельные таблицы связей.
9. Через pgAdmin создайте таблицы либо вручную, либо через SQL-скрипт, но итоговая работа с моделью должна быть видна именно в pgAdmin.
10. Загрузите тестовые данные из учебного датасета.
11. Проверьте, что по базе можно выполнить базовые запросы:
   - найти все модули системы;
   - найти владельца сервиса;
   - найти рабочую группу по системе;
   - получить список КЕ для выбранной услуги;
   - найти технологические окна для планируемого изменения.
12. Подготовьте короткое пояснение, какие элементы этой БД позже отражаются в Jira через `Components` и `select list`, а какие остаются только в модели данных.

## Ограничения задания

- Не используйте контейнеры.
- Не превращайте задание в сложное DevOps-упражнение.
- Важна простая локальная установка и понятная работа через pgAdmin.
- Цель задания не в администрировании PostgreSQL, а в том, чтобы джун понял, как модель данных превращается в реальные таблицы и связи.

## Cloud-акцент

- Это задание выполняется локально и не является частью настройки Jira.
- Связь с Jira здесь логическая: участник должен понимать, какие данные из CMDB-модели могли бы использоваться процессами Jira Service Management.

## Артефакты

- Локально установленная PostgreSQL.
- Подключение к базе через pgAdmin.
- Набор таблиц CMDB.
- Тестовые данные.
- Набор SQL-запросов для проверки модели.

## Критерии приемки

- База данных развернута локально без контейнеров.
- CMDB-модель перенесена в реальные таблицы с ключами и связями.
- Участник умеет показать структуру БД в pgAdmin и выполнить базовые запросы к ней.
