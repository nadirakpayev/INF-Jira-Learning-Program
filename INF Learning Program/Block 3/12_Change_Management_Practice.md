# Практическое задание 12. Процесс управления изменениями

## Ситуация

Нужно реализовать в Jira Service Management Cloud процесс RFC с типами изменений, несколькими этапами согласования, автоматическим созданием подзадач, контролем дат, технологических окон и пост-реализационной проверкой.

## Цель

Собрать Cloud-процесс change management, который покрывает типовые сценарии Normal, Standard и Emergency.

## Роль в программе

Это завершающее процессное задание блока 3. Оно должно завершать сквозную цепочку: инцидент или проблема -> анализ -> изменение -> дальнейший контроль.

## Что нужно сделать

1. Создайте issue type или request type для RFC.
2. Настройте типы изменений:
   - Нормальное;
   - Стандартное;
   - Срочное.
3. Настройте lifecycle основного RFC:
   - Регистрация;
   - Согласование руководителя;
   - Рассмотрение Change Manager;
   - Согласование;
   - Запланировано;
   - Выполнение;
   - Тестирование;
   - Решено;
   - Закрыто;
   - Неуспешно;
   - Отмена.
4. Создайте подзадачи:
   - Выполнение;
   - Тестирование;
   - PIR.
5. Настройте поля RFC:
   - тип изменения;
   - component;
   - модуль;
   - плановая дата начала;
   - плановая дата окончания;
   - фактическая дата начала;
   - фактическая дата окончания;
   - требуется простой;
   - план выполнения;
   - план тестирования;
   - план отката;
   - группа change manager;
   - согласующие;
   - технологическое окно.
6. Реализуйте Cloud-логику согласований:
   - руководитель инициатора;
   - Change Manager;
   - CAB или ECAB.
7. Реализуйте автоматику:
   - формирование темы RFC;
   - автозаполнение группы Change Manager по системе;
   - создание подзадач;
   - переход в выполнение по плановой дате;
   - переход в тестирование после выполнения;
   - перевод в решено после успешного тестирования;
   - создание PIR при неуспешном выполнении или выходе за рамки правил;
   - отмена связанных подзадач при отмене RFC.
8. Продумайте Cloud-вариант проверки технологического окна:
   - через select list, mapping table и automation;
   - или через внешнюю таблицу соответствий и явную валидацию после создания.
9. Настройте SLA:
   - подготовка изменения;
   - выполнение;
   - приемка инициатором.
10. Настройте очереди и представления:
   - на согласовании;
   - запланированные на неделю;
   - в выполнении;
   - в тестировании;
   - неуспешные;
   - ожидающие приемки.
11. Проведите 4 тестовых сценария:
   - стандартное изменение по шаблону;
   - нормальное изменение с ручным набором подзадач;
   - срочное изменение с post-factum согласованием;
   - неуспешное изменение с созданием PIR.

## Cloud-акцент

- Стройте процесс вокруг native approvals, automation и company-managed workflow.
- Components используйте как верхний уровень технической классификации RFC, а модуль и технологическое окно храните в справочниках select list.
- Часть серверных предикатов и валидаций в Cloud придется реализовать как пост-проверки с уведомлением и переводом статуса.
- Если change calendar доступен в вашем шаблоне проекта, включите его в демонстрацию. Если нет, сделайте эквивалент через фильтр, dashboard и календарное представление.

## Артефакты

- Настроенный RFC-процесс.
- Набор подзадач и automation-правил.
- Описание проверки технологического окна.
- Протокол тестов по четырем сценариям.

## Критерии приемки

- Основной RFC проходит через регистрацию, согласование, выполнение, тестирование и закрытие.
- Тип изменения влияет на состав подзадач и порядок работы.
- Неуспешное выполнение не теряется, а переводится в управляемый сценарий с PIR.
