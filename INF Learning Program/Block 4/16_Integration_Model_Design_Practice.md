# Практическое задание 16. Проектирование интеграционных моделей

## Ситуация

На этом этапе не требуется реализовывать интеграции технически. Но уже нужно уметь спроектировать, какие данные будут обмениваться между Jira и внешними системами, в каком виде, по каким ключам и с какими правилами обновления.

## Цель

Подготовить модель будущих интеграций на уровне сущностей, атрибутов, идентификаторов и направлений потоков данных.

Если на этом этапе не хватает понимания того, как взаимодействуют клиент, сервер, браузер и веб-приложение, сначала или параллельно проработайте [Developers Theory Fundamentals/README.md](../Developers%20Theory%20Fundamentals/README.md). Для этого задания такой трек становится `обязательным`, если без него вы не можете объяснить, откуда приходят данные, куда они передаются и почему системы вообще могут обмениваться ими по сети.

## Что нужно сделать

1. Определите список интеграционных контуров, которые логически потребуются в будущем:
   - каталог сотрудников;
   - каталог систем или сервисов;
   - CMDB или иной справочник конфигурационных единиц;
   - мониторинг;
   - внешняя аналитика или BI;
   - почта или канал уведомлений.
2. Для каждого контура опишите:
   - источник данных;
   - приемник данных;
   - сущности обмена;
   - ключевые атрибуты;
   - идентификатор записи;
   - направление обмена;
   - частоту обновления;
   - правила обработки ошибок.
3. Составьте mapping-модель:
   - внешняя сущность;
   - объект в Jira;
   - тип поля в Jira;
   - правило трансформации;
   - обязательность;
   - источник истины.
4. Отдельно опишите, какие данные:
   - создаются в Jira;
   - только читаются Jira;
   - должны обновляться двусторонне;
   - не должны редактироваться вручную после интеграции.
5. Подготовьте 2-3 интеграционных сценария:
   - синхронизация сотрудников;
   - получение данных о значительном инциденте из мониторинга;
   - выгрузка данных по доступности или изменениям во внешнюю аналитику.
6. Для каждого сценария опишите риски:
   - дубликаты;
   - потеря связей;
   - устаревшие значения;
   - отсутствие единого идентификатора;
   - конфликт ручного и автоматического обновления.

## Cloud-акцент

- Это задание не про техническую реализацию API.
- Ценность задания в том, чтобы участник умел заранее думать про сущности, ключи, источник истины и границы ответственности систем.
- Jira здесь рассматривается как одна из систем в ландшафте, а не как единственное место хранения всех данных.

## Артефакты

- Каталог будущих интеграций.
- Mapping-таблица.
- Список идентификаторов и ключей.
- Перечень рисков и проектных ограничений.

## Критерии приемки

- По модели понятно, какие данные и куда должны передаваться.
- Есть явное разделение между источником истины и целевой системой.
- Модель пригодна как основа для будущего технического задания на интеграции.
