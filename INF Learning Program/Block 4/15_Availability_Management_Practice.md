# Практическое задание 15. Процесс управления доступностью

## Ситуация

Организации нужен единый реестр недоступности, куда попадают плановые простои из изменений, внеплановые простои из значительных инцидентов и ручные записи менеджера процесса. Эти данные потом используются для отчетности и аналитики.

## Цель

Настроить в Jira Cloud управляемый процесс учета недоступности сервисов и модулей.

## Роль в программе

Это опциональное задание блока 4. Его имеет смысл выполнять после того, как уже настроены процессы изменений и значительных инцидентов.

## Что нужно сделать

1. Создайте отдельный тип записи для событий недоступности.
2. Настройте поля:
   - component;
   - недоступные модули;
   - тип недоступности;
   - тип простоя;
   - процент недоступности;
   - дата начала простоя;
   - дата окончания простоя;
   - длительность;
   - источник выявления;
   - влияние на бизнес;
   - корневая причина;
   - владелец сервиса;
   - ссылка на исходный RFC или значительный инцидент.
3. Настройте lifecycle:
   - Открыто;
   - Закрыто.
4. Реализуйте логику автоматического создания:
   - из RFC, если указано, что изменение требует простоя;
   - из значительного инцидента после завершения;
   - отдельно по каждому модулю, если это принято в вашей модели.
5. Реализуйте ручное создание записи через форму.
6. Настройте автоматические расчеты:
   - длительность в минутах;
   - тип простоя по заполненности модулей;
   - тип недоступности по источнику.
7. Если во внешней модели данных CMDB есть атрибут веса модуля, опишите как он будет использоваться в аналитике доступности.
8. Настройте реестр и фильтры:
   - все события недоступности;
   - плановые;
   - внеплановые;
   - ручные;
   - по сервису;
   - по владельцу.
9. Подготовьте Cloud-вариант отчетности:
   - dashboard;
   - экспорт данных;
   - при наличии отдельной аналитики опишите схему передачи.
10. Проведите 3 теста:
   - событие из RFC;
   - событие из значительного инцидента;
   - ручная регистрация частичной недоступности.

## Cloud-акцент

- Не пытайтесь реализовать полноценный корпоративный BI внутри Jira, если платформа для этого не подготовлена. Для учебной программы достаточно качественного реестра, фильтров, расчетных полей и понятной схемы экспорта.
- Components используйте для привязки записи недоступности к системе или сервису, а модули храните как select list.
- Если процент недоступности и вес модуля используются только для отчетности, четко разделите, что считается в Jira, а что рассчитывается во внешней аналитике.

## Артефакты

- Реестр недоступности.
- Набор полей и правил расчета.
- Сценарии автоматического создания записей.
- Описание интеграции с аналитикой.

## Критерии приемки

- Событие недоступности можно зарегистрировать вручную и автоматически.
- По записи видно источник, длительность и уровень воздействия.
- Реестр пригоден для последующей отчетности по доступности.
