# Практическое задание 5. SLA и очереди для базовых процессов

## Ситуация

После настройки сервисного каталога, инцидентов и сервисных запросов нужно привести операционный контур в рабочее состояние. Для этого недостаточно одних workflow: команде нужны понятные очереди для ежедневной работы и SLA для контроля сроков.

## Цель

Настроить SLA и очереди для процессов блока 1 так, чтобы агент мог работать с потоком заявок без ручного поиска и без потери контроля по срокам.

## Роль в программе

Это завершающее прикладное задание блока 1 перед финальным workshop. Оно использует процессы инцидентов и сервисных запросов, уже настроенные в предыдущих заданиях.

## Что нужно сделать

1. Определите перечень SLA для инцидентов:
   - первая реакция;
   - время решения;
   - отдельные цели по приоритету.
2. Определите перечень SLA для сервисных запросов:
   - первая реакция;
   - время исполнения;
   - отдельные цели по типу услуги или срочности.
3. Настройте старт, паузу и остановку SLA:
   - старт при создании или при входе в рабочий статус;
   - пауза на статусах ожидания информации;
   - остановка при решении или закрытии.
4. Подготовьте рабочие очереди для инцидентов:
   - новые;
   - моя линия;
   - высокий приоритет;
   - ожидает ответа заявителя;
   - у вендора;
   - просроченные.
5. Подготовьте рабочие очереди для сервисных запросов:
   - новые;
   - на согласовании;
   - в работе;
   - ожидает ответа заявителя;
   - просроченные;
   - решенные сегодня.
6. Настройте отображение полей в очередях:
   - ключ;
   - тема;
   - request type;
   - component;
   - приоритет;
   - исполнитель;
   - SLA;
   - дата создания.
7. Проверьте, как queues и SLA работают на тестовых кейсах:
   - один инцидент с эскалацией;
   - один сервисный запрос с согласованием;
   - один кейс с запросом доп. информации.

## Cloud-акцент

- Используйте native SLA Jira Service Management.
- Не создавайте слишком много очередей: они должны помогать работе, а не дробить поток без пользы.
- Настройка очередей должна отражать реальную операционную модель команды, а не просто повторять статусы workflow.

## Артефакты

- Набор SLA для инцидентов.
- Набор SLA для сервисных запросов.
- Рабочие очереди агентов.
- Результаты тестовой проверки SLA и очередей.

## Критерии приемки

- Агент видит, какие заявки требуют реакции прямо сейчас.
- Просрочки и риски по срокам видны без ручного поиска.
- Очереди и SLA логично поддерживают процессы блока 1.
