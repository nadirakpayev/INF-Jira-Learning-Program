# Практическое задание 4. Процесс запросов на обслуживание

## Ситуация

В учебной среде нужно реализовать процесс сервисных запросов: заявитель выбирает услугу, при необходимости проходит согласование, запрос маршрутизируется на нужную линию поддержки, исполняется, а затем закрывается.

## Цель

Настроить в Jira Service Management Cloud управляемый процесс запросов на обслуживание с понятным порталом, SLA и базовой автоматизацией.

## Роль в программе

Это третье сквозное задание блока 1. Оно должно использовать тот же каталог услуг, `Components` и правила маршрутизации, что и процесс инцидентов.

## Что нужно сделать

1. Создайте request type для запросов на обслуживание.
2. Спроектируйте lifecycle со статусами:
   - Открыт;
   - Согласование;
   - Назначен на 1-ю линию;
   - Назначен на 2-ю линию;
   - Назначен на 3-ю линию;
   - В работе на 1-й линии;
   - В работе на 2-й линии;
   - В работе на 3-й линии;
   - Запрос доп. информации;
   - Смена линии;
   - Решен;
   - Закрыт.
3. Определите набор полей:
   - категория;
   - услуга;
   - описание;
   - component;
   - модуль или КЕ;
   - приоритет;
   - инициатор;
   - группа исполнителей;
   - координатор;
   - исполнитель;
   - решение;
   - вложения.
4. Настройте формы:
   - создание через портал;
   - создание агентом;
   - запрос дополнительной информации;
   - решение;
   - закрытие.
5. Реализуйте согласование для тех услуг, где оно обязательно.
6. Настройте автоматизации:
   - формирование темы заявки;
   - автозаполнение группы поддержки;
   - заполнение или валидация component и select list-справочников;
   - автопереход в нужный статус после согласования;
   - напоминание инициатору в статусе ожидания информации;
   - автозакрытие после решения через согласованный срок.
7. Настройте очереди:
   - все открытые запросы;
   - заявки на согласовании;
   - 1-я линия;
   - 2-я линия;
   - 3-я линия;
   - ожидает ответа заявителя;
   - решенные сегодня;
   - просроченные по SLA.
8. Настройте SLA:
   - первая реакция;
   - время исполнения;
   - отдельные цели по приоритету или типу услуги.
9. Подготовьте 3 тестовых сценария:
   - запрос без согласования;
   - запрос с согласованием;
   - запрос с переводом между линиями.

## Cloud-акцент

- Для согласований используйте native approvals Jira Service Management.
- Для скрытия и условной логики полей используйте forms и логику request type, а не серверные behaviours.
- Components используйте для систем, сервисов или зон ответственности, а детальные справочники держите в select list.
- Если одна и та же услуга имеет разный путь исполнения, фиксируйте это через automation и очереди, а не через чрезмерно сложный workflow.

## Артефакты

- Настроенный request type.
- Workflow и карта переходов.
- Набор automation-правил.
- SLA и очереди.
- Результаты тестовых прогонов.

## Критерии приемки

- Запрос можно создать через портал и через агентский интерфейс.
- Согласование, если требуется, влияет на маршрут заявки.
- Очереди показывают актуальную картину работы команды.
- SLA отслеживается нативно и не требует внешнего плагина.
